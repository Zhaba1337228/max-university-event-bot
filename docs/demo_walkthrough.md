# Demo walkthrough — MAX University Event Bot

Пошаговый сценарий для жюри хакатона и презентации MVP. Длительность
живой демонстрации — 7–10 минут. Цель — показать три ключевых сценария
(абитуриент в боте, организатор в боте, веб-админка) и продемонстрировать
graceful degradation при отсутствии живых внешних сервисов (GigaChat).

> Связанные документы: README.md (быстрый старт), [`runbook.md`](./runbook.md)
> (эксплуатация), [`progress.md`](./progress.md) (статус MVP по плану).

---

## 0. Минимальный `.env` для запуска

Скопировать `.env.example` в `.env` и заполнить **минимум** четыре переменные:

```bash
# Обязательно
MAX_BOT_TOKEN=<токен из кабинета MAX / MasterBot>
ADMIN_SESSION_KEY=<openssl rand -base64 32 | tr -d '\n'>   # ≥ 32 символов
DATABASE_URL=postgres://app:app@postgres:5432/maxbot?sslmode=disable

# Для AI (опционально, без него работает с fallback'ом)
GIGACHAT_AUTH_KEY=<base64 client_id:client_secret из ЛК GigaChat>

# Чтобы /organizer был доступен — добавить свой max_user_id
ORGANIZER_USER_IDS=<ваш user_id из MAX>
```

Запуск всего стека (postgres + migrate + bot + web):

```bash
make docker-up
```

После старта:
- бот доступен в MAX по `/start`;
- веб-админка — на `http://localhost:3000`;
- admin REST API — `http://localhost:8081` (через rewrite в Next.js `/api/*`);
- healthcheck — `GET http://localhost:8080/healthz`.

Команда `make docker-up` поднимает `postgres → migrate (одноразово) → bot → web`
в правильном порядке (см. `deployments/docker-compose.yml`, depends_on +
`service_healthy` / `service_completed_successfully`). На свежей БД миграция
`20260101000010_seed_demo_event.sql` создаёт три демо-события.

## 1. Сценарий абитуриента в боте

Цель: пройти полный путь записи на событие и получить QR-приглашение.

1. **`/start`** — бот приветствует, показывает главное меню:
   «Записаться», «Мои записи», «AI-подбор», «Помощь».
2. **«Записаться»** → бот выводит постранично события, открытые для
   регистрации (`status = 'open'`, `starts_at > now()`,
   keyboard pageSize = 8). Демо-сидов три: «День открытых дверей ИТ-направлений»,
   «Консультация по поступлению», «Онлайн-знакомство с программной инженерией».
3. **Тап по событию** → карточка с описанием, датой (Europe/Moscow),
   местом, форматом, capacity и числом свободных мест. Кнопка «Записаться».
4. **Согласие на ПДн** (152-ФЗ). Если у пользователя нет согласия —
   показывается экран consent с двумя кнопками: «Согласен» / «Отказаться».
   Версия политики фиксируется в `users.consent_policy_ver`
   (`PRIVACY_POLICY_VERSION`).
5. **ФИО** — текстовый ввод. Валидация: достаточно полное имя одним
   сообщением.
6. **Направление / интерес** — короткий текст (≤ 200 символов).
   Сохраняется в `registrations.interest_program`, используется в
   AI-сводке организатору и в рекомендациях.
7. **Подтверждение** — бот эхом показывает мероприятие, ФИО и направление.
   Кнопки «Подтвердить» / «Изменить» / «Отмена».
8. **«Подтвердить»** → сервис создаёт запись `registrations(status='registered')`
   либо `waitlist` (если capacity исчерпана и `WAITLIST_ENABLED=true`),
   генерирует attendance code (uuid) и **отправляет QR-картинку** прямо в
   чат (`SendWithResult` + `qr_sent_message_id`). Также бот логирует
   действие в `action_logs` и отправляет шаблон «registration_created».
9. **После регистрации** пользователь видит короткий код записи в тексте
   подтверждения и позже может снова открыть его через «Моя запись».
10. **Напоминания** — scheduler создаёт `reminder_24h` и `reminder_1h`,
    если до события хватает времени. Если до старта уже меньше 24 часов,
    будет только часовое напоминание.
11. **«Мои записи» из главного меню** → список активных регистраций.
    Кнопки «Показать мой QR», «Отменить запись», «История».
12. **«Отменить запись»** → двухшаговое подтверждение (FSM-guard защищает
    от устаревших inline-кнопок). Если у события включён лист ожидания,
    сразу промоутится первый из waitlist'а (см. `service/registration.go`).
13. **`/forget_me`** → бот показывает предупреждение и кнопку
    «Подтвердить удаление». При подтверждении —
    `repo.Users.ForgetMe(...)` каскадно удаляет регистрации, FSM-state,
    action_logs (FK ON DELETE SET NULL/ CASCADE), затем сам user-row.
    Это требование 152-ФЗ.

## 2. Сценарий организатора в боте

Цель: посмотреть участников события и сделать рассылку с AI-улучшением.

> Чтобы получить роль organizer на демо — добавить свой `max_user_id`
> в `ORGANIZER_USER_IDS` (CSV) и перезапустить бота. RBAC применяется
> при первом hit'е `/organizer` (см. `service/auth.go`).

1. **`/organizer`** — меню организатора: «Мои события», «Создать событие».
2. **«Мои события»** → список своих событий с краткой статистикой
   (registered / waitlist / capacity / attended).
3. **Тап по событию** → карточка с расширенной сводкой
   и клавиатура `[Участники | CSV | Рассылка | Закрыть | AI-сводка]`.
4. **«Участники»** → постраничный список ФИО + контакт + интерес + статус.
5. **«CSV»** → бот высылает CSV-текст одним или несколькими сообщениями
   (`organizer_list.go::exportCSV`, escape по RFC 4180). MAX-лимит 4000
   символов на сообщение, поэтому большие выгрузки делаем через
   веб-админку (`/events/[id]/participants`, GET `/api/events/:id/participants`).
6. **«Рассылка»** → бот спрашивает текст. После ввода — экран превью с
   кнопками `[Отправить | AI-улучшить | Отмена]`.
7. **«AI-улучшить»** → бот вызывает `gigachat.RewriteNotification`,
   показывает новый вариант. На ошибку (`ErrAIUnavailable`) оставляет
   исходный текст с пометкой «AI недоступен».
8. **«Отправить»** → создаются строки в `notifications` с типом
   `organizer_broadcast` и `status='pending'`. Scheduler-job
   `dispatchDue` (раз в минуту) разошлёт их с rate-limit'ом
   `NOTIFICATION_RATE_LIMIT_RPS=20`.
9. **«AI-сводка»** → `gigachat.OrganizerSummary` агрегирует интересы
   зарегистрированных и строит краткий саммари. При недоступности —
   fallback на детерминированный top-N интересов + сообщение
   «AI недоступен» (см. ниже §4).

## 3. Сценарий веб-админки

Цель: показать magic-link логин, дашборд, операции с событием и QR-checkin.

1. **В боте: `/admin_login`** → бот выдаёт ссылку вида
   `http://localhost:3000/auth?t=<magic-jwt>` (ключ — `ADMIN_SESSION_KEY`,
   TTL короткий, см. `service/auth.go`).
2. **Клик по ссылке** → `/auth` парсит `t`, делает
   `POST /api/auth/exchange`, бэкенд ставит httpOnly cookie `session_jwt`,
   фронт редиректит на `/dashboard`. Все API-запросы идут через Next.js
   rewrite `/api/* → API_UPSTREAM` (см. `web/next.config.mjs`),
   поэтому cookie остаются same-origin (важно для SameSite=Strict).
3. **`/dashboard`** — список своих событий с краткой статистикой.
4. **`/events/[id]`** — карточка события с:
   - агрегированной статистикой (`events.Stats`);
   - кнопками `[Участники | Рассылка | Закрыть | Check-in]`;
   - AI-сводкой (если включена и GigaChat доступен).
5. **`/events/[id]/participants`** — таблица всех регистраций с поиском
   и фильтрами по статусу.
6. **`/events/[id]/broadcast`** — форма рассылки с предпросмотром и
   кнопкой «AI-улучшить» (тот же `service.AI.RewriteNotification`).
7. **`/checkin`** — страница QR check-in:
   - в браузере с камерой (`@yudiel/react-qr-scanner`) — сканирование с
     телефона/вебкамеры в реальном времени;
   - fallback: ручной ввод attendance code;
   - на каждый успешный скан — `POST /api/checkin`, статус регистрации
     меняется на `attended`, в боте у участника появляется отметка в
     «Моих записях».

## 4. GigaChat выключен — что увидит жюри

`GIGACHAT_AUTH_KEY=""` — это поддерживаемый сценарий (by design).
GigaChat-клиент создаётся только при заполненном ключе; в остальных
местах AI-фасад возвращает `ErrAIUnavailable`, и UI деградирует:

- **«AI-подбор»** в боте — fallback на обычный список открытых событий.
- **«AI-сводка»** организатору — fallback на детерминированный top-N
  интересов из `registrations.interest` + сообщение «AI недоступен».
- **«AI-улучшить»** рассылку — оставляет исходный текст и помечает,
  что AI недоступен.
- **Веб-админка** — AI-блоки скрываются или показывают плейсхолдер.

Это позволяет проводить демо без живого ключа (например, на изолированном
стенде или при отсутствии сети до Sber API).

## 5. Known issues и ограничения MVP

- **CSV в боте — текстовый, не attachment.** MAX-лимит 4000 символов
  на сообщение → крупные выгрузки делаем в веб-админке.
- **Long-poll + webhook не работают одновременно.** Режим переключается
  через `MAX_BOT_MODE=longpoll|webhook`. В webhook-режиме MAX через
  8 ч простоя отписывает нас от подписки; на старте мы вызываем
  `ensureSubscription` (`GetSubscriptions` → `Subscribe`).
- **Bootstrap-роли через ENV.** Первая разметка organizer/admin делается
  через `ORGANIZER_USER_IDS` / `ADMIN_USER_IDS` (CSV `max_user_id`).
  Перевыдача роли через UI — out of scope MVP.
- **Аутентификация stateless.** JWT в httpOnly cookie, серверной таблицы
  сессий нет. Ротация `ADMIN_SESSION_KEY` инвалидирует все сессии разом
  (см. `runbook.md` §«Ротация секретов»).
- **GigaChat self-signed TLS на dev.** `GIGACHAT_INSECURE_TLS=true`
  допустим только локально. В docker-compose значение приходит через
  `.env`; в prod должно быть `false`.
- **На демо не показываем `/forget_me`.** Команда работает и
  каскадно удаляет данные, но в живой среде это разрушительная операция —
  лучше тестировать на отдельном `max_user_id`.

## 6. Backup-план: видео-демо

Если живая среда не поднимется на демо (нет интернета до MAX API,
заблокирован GigaChat, упал docker-host) — переключаемся на заранее
записанный screen-cast того же сценария §1–§3. Видео-файл хранится у
организатора презентации; команда `make docker-up` всё равно
выполняется на сцене как proof — даже без MAX_BOT_TOKEN миграции
поднимутся и `healthz` ответит 200.
