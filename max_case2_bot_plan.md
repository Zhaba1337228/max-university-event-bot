# План разработки чат-бота MAX для записи абитуриента на мероприятие университета

## 0. Выбранный кейс

**Кейс №2: Запись абитуриента на мероприятие университета.**

Формат решения: **чат-бот в MAX без мини-приложения**.

Цель: сделать понятный сервис, который позволяет абитуриенту быстро записаться на конкретное мероприятие университета, получить подтверждение, видеть статус записи, отменить участие, а организаторам — управлять списком участников и рассылками по одному мероприятию.

---

## 1. Ключевая идея проекта

Бот закрывает один конкретный сценарий:

> Абитуриент открывает бота → выбирает мероприятие → вводит данные → получает подтверждение → может проверить статус или отменить запись → организатор видит список участников и отправляет важные уведомления.

Фокус не на «супер-платформе для всего университета», а на **очень отполированном точечном сервисе**, который реально можно внедрить.

---

## 2. Пользователи и роли

### 2.1. Абитуриент

Возможности:

- посмотреть доступные мероприятия;
- узнать краткую информацию о мероприятии;
- записаться;
- получить подтверждение записи;
- проверить статус;
- отменить запись;
- получить напоминание;
- получить уведомление об изменении времени/места/формата.

### 2.2. Организатор мероприятия

Возможности:

- создать или выбрать мероприятие;
- посмотреть список записавшихся;
- посмотреть статистику;
- выгрузить список участников;
- отправить уведомление всем участникам;
- закрыть запись;
- вручную отменить или подтвердить участника при необходимости.

### 2.3. Администратор

Возможности:

- управлять организаторами;
- смотреть все мероприятия;
- менять настройки бота;
- просматривать историю действий.

## 3. MVP-функциональность

### 3.1. Для абитуриента

Обязательный минимум:

- стартовое меню;
- список доступных мероприятий;
- карточка мероприятия;
- запись на мероприятие;
- подтверждение записи;
- просмотр статуса;
- отмена записи;
- история действий;
- уведомление/напоминание.

### 3.2. Для организатора

Обязательный минимум:

- просмотр активного мероприятия;
- список участников;
- количество мест;
- количество записей;
- отправка объявления участникам;
- закрытие регистрации.

### 3.3. Для защиты от хаоса

- проверка повторной записи;
- ограничение по количеству мест;
- понятные статусы;
- логирование действий;
- подтверждение опасных действий: отмена, закрытие регистрации, массовая рассылка.

## 4. Статусы записи

Рекомендуемые статусы:

| Статус | Значение |
|---|---|
| `draft` | пользователь начал запись, но не завершил |
| `registered` | пользователь успешно записан |
| `waitlist` | мест нет, пользователь в листе ожидания |
| `cancelled_by_user` | пользователь сам отменил запись |
| `cancelled_by_organizer` | запись отменил организатор |
| `attended` | пользователь пришел на мероприятие |
| `no_show` | пользователь не пришел |

Для MVP достаточно:

- `registered`;
- `waitlist`;
- `cancelled_by_user`;
- `cancelled_by_organizer`.

## 5. Основной пользовательский сценарий

### 5.1. Первый запуск

Пользователь нажимает `/start`.

Бот отвечает:

```text
Привет! Я помогу записаться на мероприятие университета.

Что хотите сделать?
```

Кнопки:

- `Записаться на мероприятие`;
- `Моя запись`;
- `Отменить запись`;
- `Помощь`.

### 5.2. Выбор мероприятия

Бот показывает список:

```text
Доступные мероприятия:

1. День открытых дверей ИТ-направлений
   25 сентября, 15:00
   Осталось мест: 42

2. Консультация по поступлению
   27 сентября, 12:00
   Осталось мест: 18
```

Кнопки:

- `День открытых дверей ИТ`;
- `Консультация по поступлению`;
- `Назад`.

### 5.3. Карточка мероприятия

```text
День открытых дверей ИТ-направлений

Дата: 25 сентября
Время: 15:00
Место: Главный корпус, аудитория 301
Формат: очно
Осталось мест: 42

На мероприятии расскажем о программах, вступительных испытаниях, бюджетных местах и карьерных траекториях.
```

Кнопки:

- `Записаться`;
- `Назад к списку`;
- `Задать вопрос`.

### 5.4. Сбор данных

Минимальные поля:

- ФИО;
- телефон;
- email;
- интересующее направление;
- согласие на обработку данных.

Лучше не перегружать форму. Для демо достаточно:

1. ФИО;
2. телефон или email;
3. направление интереса.

### 5.5. Подтверждение

```text
Проверьте данные:

Мероприятие: День открытых дверей ИТ-направлений
Дата: 25 сентября, 15:00
ФИО: Иванов Иван Иванович
Контакт: ivan@example.com
Направление: Прикладная информатика

Все верно?
```

Кнопки:

- `Подтвердить`;
- `Изменить данные`;
- `Отменить`.

### 5.6. Успешная запись

```text
Вы записаны на мероприятие.

Мероприятие: День открытых дверей ИТ-направлений
Дата: 25 сентября, 15:00
Место: Главный корпус, аудитория 301
Статус: запись подтверждена

За день до мероприятия я пришлю напоминание.
```

Кнопки:

- `Моя запись`;
- `Отменить запись`;
- `В главное меню`.

## 6. Сценарий отмены записи

```text
Вы действительно хотите отменить запись?

Мероприятие: День открытых дверей ИТ-направлений
Дата: 25 сентября, 15:00
```

Кнопки:

- `Да, отменить`;
- `Нет, оставить запись`.

После отмены:

```text
Запись отменена.

Если планы изменятся, вы сможете записаться снова, если останутся свободные места.
```

Если есть лист ожидания, первое место автоматически переходит следующему участнику.

## 7. Сценарий листа ожидания

Если мест нет:

```text
Свободных мест сейчас нет.

Можем добавить вас в лист ожидания. Если место освободится, вы получите уведомление.
```

Кнопки:

- `Встать в лист ожидания`;
- `Выбрать другое мероприятие`;
- `Назад`.

Если место освободилось:

```text
Освободилось место на мероприятие:
День открытых дверей ИТ-направлений.

Хотите подтвердить участие?
```

Кнопки:

- `Подтвердить участие`;
- `Отказаться`.

## 9. История действий

Для прозрачности бот хранит события:

- пользователь начал запись;
- пользователь подтвердил запись;
- пользователь отменил запись;
- организатор отправил уведомление;
- организатор закрыл регистрацию;
- пользователь переведен из листа ожидания в участники.

Пример для абитуриента:

```text
История вашей записи:

12.09 14:20 — запись создана
12.09 14:21 — запись подтверждена
24.09 10:00 — отправлено напоминание
```

## 10. Архитектура без мини-приложения

### 10.1. Общая схема

```text
MAX Messenger
     |
     v
MAX Bot API / Webhook
     |
     v
Backend приложения
     |
     +--> FSM / сценарии диалогов
     +--> Бизнес-логика записи
     +--> База данных
     +--> Планировщик уведомлений
     +--> Админские команды
```

### 10.2. Рекомендуемый стек

Вариант для быстрого хакатона:

- **Backend:** Python + FastAPI;
- **Bot logic:** отдельный слой handlers/services;
- **Database:** PostgreSQL;
- **ORM:** SQLAlchemy или SQLModel;
- **Migrations:** Alembic;
- **Scheduler:** APScheduler или Celery Beat;
- **Deployment:** Docker Compose;
- **Logs:** stdout + структурированные события;
- **Docs:** README + схема сценариев.

Альтернатива:

- Node.js + NestJS/Fastify;
- PostgreSQL;
- Prisma;
- BullMQ для очередей.

Главное: не усложнять. Для 17 дней лучше взять стек, который команда знает лучше.

## 13. UX-принципы

### 13.1. Короткие сообщения

Не писать полотна текста. Одно сообщение — одно действие.

### 13.2. Кнопки вместо ручного ввода

Ручной ввод нужен только там, где без него нельзя:

- ФИО;
- контакт;
- направление интереса;
- текст рассылки от организатора.

### 13.3. Всегда показывать статус

Пользователь не должен думать:

- записался он или нет;
- где пройдет мероприятие;
- можно ли отменить запись;
- что будет дальше.

### 13.4. Все опасные действия подтверждать

Подтверждение нужно для:

- отмены записи;
- массовой рассылки;
- закрытия регистрации;
- удаления мероприятия.

## 14. Что показать на демо

Лучший демо-сценарий:

1. Абитуриент открывает бота.
2. Выбирает мероприятие.
3. Видит карточку мероприятия.
4. Заполняет данные.
5. Получает подтверждение.
6. Проверяет статус записи.
7. Организатор открывает панель через команды бота.
8. Видит нового участника в списке.
9. Отправляет уведомление всем участникам.
10. Абитуриент получает уведомление.
11. Абитуриент отменяет запись.
12. Организатор видит обновленную статистику.

Это выглядит законченным продуктом, даже без мини-приложения.

### 15.1. Прозрачная история

У каждого действия есть история: кто, когда и что сделал.

### 15.2. Лист ожидания

Организатор может информировать только участников конкретного мероприятия.

Пользователь всегда понимает, что происходит с его заявкой.

### 15.5. Антидубли

### 15.6. Готовность к реальному внедрению

Есть роли, база данных, история действий, ограничения мест и сценарии отмены.

## 16. AI-фишки через GigaChat

ИИ лучше добавлять не как «чат ради чата», а как усиление основного сценария записи. На защите это можно подать так:

> Мы используем GigaChat не вместо понятного интерфейса, а как интеллектуальный слой, который помогает абитуриенту быстрее выбрать подходящее мероприятие, а организатору — меньше времени тратить на рутину.

### 16.1. Главная AI-фишка: умный помощник абитуриента

Сценарий:

Пользователь не знает, на какое мероприятие идти. Он пишет обычным языком:

```text
Я хочу поступать на программирование, но не понимаю, что выбрать.
```

Бот через GigaChat анализирует запрос и отвечает:

```text
Похоже, вам подойдут мероприятия по ИТ-направлениям.

Рекомендую:

1. День открытых дверей ИТ-направлений
   Подойдет, если хотите узнать про программы и поступление.

2. Консультация по ЕГЭ и вступительным испытаниям
   Подойдет, если хотите понять требования и проходные баллы.

Хотите записаться на первое мероприятие?
```

Кнопки:

- `Записаться`;
- `Показать подробнее`;
- `Выбрать другое`;
- `Задать вопрос`.

Почему это сильно:

- абитуриенту не надо читать длинный список;
- бот выглядит умнее обычной формы;
- ИИ решает реальную проблему выбора.

### 16.2. AI-вопросы по мероприятию

После карточки мероприятия пользователь может нажать:

```text
Задать вопрос о мероприятии
```

Примеры вопросов:

```text
Что там будет?
Мне это подойдет, если я сдаю информатику?
Можно прийти с родителями?
Что взять с собой?
```

GigaChat отвечает только на основе данных мероприятия и заранее подготовленной базы FAQ.

Важно: не давать ИИ выдумывать правила поступления. Лучше использовать формат:

```text
По описанию мероприятия...
```

или:

```text
В информации мероприятия это не указано. Я могу передать вопрос организатору.
```

### 16.3. AI-суммаризация мероприятия

Организатор вводит длинное описание:

```text
На мероприятии представители приемной комиссии расскажут...
```

GigaChat делает короткую версию для карточки бота:

```text
Кратко:
Узнаете про ИТ-направления, бюджетные места, вступительные испытания и карьерные перспективы.
```

Польза:

- карточки мероприятий становятся короткими;
- организатору не надо вручную адаптировать текст под чат;
- демо выглядит аккуратнее.

### 16.4. AI-помощник для рассылок

Организатор пишет сырой текст:

```text
завтра все приходите в 15 аудитория 301 не опаздывайте
```

Бот предлагает улучшенную версию:

```text
Напоминаем, что мероприятие «День открытых дверей ИТ-направлений» пройдет завтра в 15:00.

Место проведения: главный корпус, аудитория 301.

Пожалуйста, приходите за 10–15 минут до начала.
```

Кнопки:

- `Отправить эту версию`;
- `Изменить`;
- `Оставить мой текст`;
- `Отмена`.

Почему это хорошо для защиты:

- ИИ снижает риск кривых рассылок;
- сообщения становятся официальными и понятными;
- организатор экономит время.

### 16.5. AI-сводка для организатора

Перед мероприятием организатор нажимает:

```text
Сводка по мероприятию
```

Бот собирает статистику и GigaChat формирует человеческий отчет:

```text
Сводка:

На мероприятие записано 78 человек из 100 возможных.
Свободно 22 места. Отменили участие 5 человек.
Больше всего участников интересуются направлениями:

1. Прикладная информатика — 31
2. Программная инженерия — 24
3. Информационная безопасность — 12

Рекомендация: отправить напоминание сегодня вечером и отдельно указать аудиторию.
```

Это очень презентационная фича: выглядит как «аналитика», но делается относительно просто.

### 16.6. AI-классификация вопросов

Если абитуриент задает вопрос, бот определяет тип:

- про поступление;
- про место проведения;
- про документы;
- про расписание;
- про отмену записи;
- другое.

Если вопрос простой — бот отвечает из FAQ.

Если вопрос сложный — бот предлагает передать организатору:

```text
Я не нашел точного ответа в информации мероприятия.
Передать вопрос организатору?
```

Так ИИ не ломает доверие и не галлюцинирует.

### 16.7. AI-персональное напоминание

Обычное напоминание:

```text
Завтра мероприятие в 15:00.
```

AI-напоминание:

```text
Иван, завтра в 15:00 пройдет День открытых дверей ИТ-направлений.

Вы указывали интерес к прикладной информатике — на встрече можно будет задать вопросы по этой программе.

Место: главный корпус, аудитория 301.
```

Главное не переборщить. Это P1/P2, но на демо смотрится приятно.

### 16.8. AI-антихаос: проверка рассылки перед отправкой

Перед массовой рассылкой GigaChat проверяет текст:

- слишком длинный;
- нет даты;
- нет времени;
- нет места;
- слишком неформальный стиль;
- есть неоднозначность.

Пример:

```text
В сообщении не указано место проведения.
Добавить аудиторию перед отправкой?
```

Это отличная фича для кейса, потому что кейс прямо говорит про «управляемое информирование».

### 16.9. Что точно стоит взять в MVP с ИИ

Минимальный, но сильный набор:

1. **AI-подбор мероприятия по интересу абитуриента**.
2. **AI-улучшение текста рассылки для организатора**.
3. **AI-сводка по мероприятию для организатора**.

Этого достаточно, чтобы на защите сказать:

> У нас не просто форма записи в боте. У нас бот помогает абитуриенту выбрать релевантное мероприятие, помогает организатору понятно коммуницировать и формирует управленческую сводку.

### 16.10. Что можно оставить как P2

- полноценный свободный AI-чат;
- сложная RAG-база по правилам поступления;
- генерация персональных образовательных рекомендаций;
- анализ настроений;
- прогноз явки.

Эти вещи звучат красиво, но могут сожрать время.

### 16.11. Безопасные ограничения для ИИ

Чтобы ИИ не навредил:

- не принимать запись через свободный текст без подтверждения кнопкой;
- не давать юридически значимые обещания;
- не выдумывать проходные баллы и правила поступления;
- отвечать только по данным мероприятия и FAQ;
- все массовые рассылки отправлять только после подтверждения организатора;
- логировать AI-ответы;
- показывать fallback: `Не уверен, передам вопрос организатору`.

### 16.12. Как красиво назвать AI-модуль

Варианты:

- `AI-ассистент абитуриента`;
- `Умный подбор мероприятия`;
- `GigaChat-помощник поступающего`;
- `AI-сводка для организатора`;
- `Умные рассылки`.

Лучшее название для презентации:

> **GigaChat-ассистент для абитуриента и организатора**

## 17. AI-архитектура

```text
MAX Bot
   |
   v
Backend
   |
   +--> обычные сценарии записи
   +--> AI Service
           |
           +--> prompt templates
           +--> FAQ / event context
           +--> GigaChat API
           +--> moderation / fallback
```

### 17.1. Где использовать GigaChat технически

Отдельный сервисный слой:

```text
services/ai/
├── gigachat_client.py
├── prompts.py
├── event_recommender.py
├── notification_rewriter.py
└── organizer_summary.py
```

### 17.2. Принцип работы

1. Бот собирает контекст: мероприятие, FAQ, вопрос пользователя.
2. Backend формирует короткий prompt.
3. GigaChat возвращает ответ.
4. Backend проверяет ответ.
5. Бот показывает ответ пользователю с кнопками действия.

### 17.3. Пример prompt для подбора мероприятия

```text
Ты помощник приемной комиссии университета.
Твоя задача — выбрать наиболее подходящее мероприятие для абитуриента.

Не выдумывай мероприятий. Используй только список ниже.

Интерес пользователя:
{user_interest}

Доступные мероприятия:
{events}

Верни 1–2 рекомендации коротко и понятно.
```

### 17.4. Пример prompt для улучшения рассылки

```text
Ты помогаешь организатору университетского мероприятия написать понятное уведомление.

Сохрани смысл, сделай текст официальным, коротким и дружелюбным.
Обязательно укажи дату, время и место, если они есть в контексте.

Исходный текст:
{draft_text}

Контекст мероприятия:
{event_context}
```

### 17.5. Пример prompt для сводки

```text
Сформируй короткую управленческую сводку по мероприятию для организатора.

Данные:
{event_stats}

Нужно:
- 3–5 коротких предложений;
- ключевые числа;
- 1 практическая рекомендация.
```

## 18. Что не делать

Чтобы не утонуть:

- не делать мини-приложение;
- не делать полноценную CRM;
- не делать сложный визуальный админский кабинет;
- не делать регистрацию на десятки типов мероприятий;
- не добавлять оплату;
- не добавлять авторизацию через внешние сервисы, если это не требуется;
- не писать огромные анкеты;
- не делать сложную систему прав;
- не пытаться заменить сайт университета.

Фокус: **одно мероприятие, понятная запись, прозрачный статус, управляемое информирование**.

## 21. Риски и как их закрыть

| Риск | Что делать |
|---|---|
| MAX Bot API окажется непривычным | в первые 3–4 дня сделать минимальный echo/start бот |
| Слишком много функций | жестко держать MVP |
| Не хватит времени на админку | делать админские функции прямо в боте |
| Бот будет неудобным | сначала написать UX-тексты, потом код |
| Демо сломается | подготовить тестовые данные и видео |
| Команда начнет делать лишнее | каждый день сверяться с P0/P1/P2 |
| GigaChat API будет недоступен | сделать fallback: обычный подбор по кнопкам и ручной текст рассылки |
| ИИ начнет выдумывать | ограничить ответы данными мероприятия и FAQ |
| AI-фичи съедят время | взять только 3 функции: подбор, улучшение рассылки, сводка |

## 24. Pitch для защиты

> Мы сделали чат-бота в MAX, который превращает запись абитуриента на мероприятие университета в короткий и прозрачный процесс. Поэтому бот не просто принимает заявки, а реально снижает коммуникационный хаос вокруг мероприятия.
